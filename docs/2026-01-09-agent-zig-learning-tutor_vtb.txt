вҸә Great question! Let's think through what FrameContext might need based on how it's
  actually used.

  Current implementation:
  const FrameContext = struct {
      pixel_buffer: ?vtb.CVPixelBufferRef,
  };

  What else might be useful?

  1. Error handling - The callback receives an OSStatus parameter. Right now you just
  print errors, but you might want to capture them so the caller knows if decode failed:
  const FrameContext = struct {
      pixel_buffer: ?vtb.CVPixelBufferRef,
      decode_status: vtb.OSStatus,  // Capture success/failure
  };

  2. Timing information - The callback receives presentationTimeStamp and
  presentationDuration. You might want these for A/V sync later:
  const FrameContext = struct {
      pixel_buffer: ?vtb.CVPixelBufferRef,
      decode_status: vtb.OSStatus,
      pts: ?vtb.CMTime,  // Actual PTS from decoder
      duration: ?vtb.CMTime,
  };

  3. Info flags - The infoFlags parameter tells you things like "frame was dropped" or
  "decode happened asynchronously":
  decode_info_flags: u32,  // VTDecodeInfoFlags

  My recommendation for Phase 6:

  For your current test code, pixel_buffer alone is fine! You're just verifying decode
  works.

  But when you build the VideoToolboxDecoder struct in Phase 6, I'd add:
  - decode_status (essential - you need to know if decode failed)
  - pts and duration (nice to have - useful for video playback)
  - decode_info_flags (optional - mainly for debugging)

  Question for you: Do you want to add error handling now, or wait until Phase 6 when you
  refactor into the decoder struct? The current code works, so you could leave it and come
   back to this when you need production-ready error handling.
